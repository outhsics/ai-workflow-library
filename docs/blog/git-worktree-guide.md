# Git Worktree 多分支并行开发指南

`git worktree` 解决的是一个很具体的问题：同一个仓库，如何同时在多个目录里操作不同分支，而不是来回 `stash`、`switch`、`pop`。

## 它最适合什么场景

- 你正在开发新功能，但临时要去主线排查线上问题
- 你需要同时对比 `main`、`develop`、`hotfix` 的页面行为
- 你不想在一个目录里反复切分支，把现场切乱

如果你的工作经常落在“开发中 + 排障中 + 验证中”三个状态之间，`worktree` 会非常值。

## 核心思路

正常情况下，一个仓库目录一次只能停在一个分支。

`worktree` 的做法是：

- 保留当前主工作目录不动
- 额外创建几个目录
- 每个目录各自检出一个分支

这样同一个仓库就能同时拥有多个独立现场。

## 最常用命令

### 基于已有分支创建

```bash
git worktree add ../repo-main main
```

### 从远程分支拉一个新分支出来

```bash
git worktree add -b fix/login-timeout ../repo-fix origin/main
```

### 查看所有 worktree

```bash
git worktree list
```

### 删除 worktree

```bash
git worktree remove ../repo-fix
git worktree prune
```

## 一个很实用的目录分法

假设你的主项目目录是 `~/project`，可以这样拆：

- `~/project`：当前开发分支
- `~/project-main`：主线，只负责对比
- `~/project-fix`：临时修复
- `~/project-test`：验证分支

这样做的好处是非常直接的：

- 不容易误提交到错误分支
- 能同时开多个本地服务做对比
- 回退和验证不会污染当前开发现场

## 我认为它真正的价值

很多人把 `worktree` 看成高级 Git 技巧，但它本质上只是“隔离现场”。

它最有价值的时刻，不是在平稳开发期，而是在：

- 快速排查 bug
- 线上问题回溯
- 多个修复策略并行实验
- 临时 hotfix 插队

这些情况下，目录隔离往往比命令技巧更重要。

## 什么时候不值得上

- 你平时就只维护一条分支
- 仓库很大，每开一个目录都要重装一套依赖且成本很高
- 团队对 Git 本身不熟，额外概念反而增加负担

这不是默认动作，而是一种在“复杂切换”场景下非常有效的工具。

## 使用时要注意的三件事

### 1. 构建产物是独立的

每个 worktree 都有自己的：

- `node_modules`
- `dist`
- `.next`
- `build`

它的优点是不会互相污染，代价是空间占用会增加。

### 2. 清理要跟上

实验性的目录用完就删，不然久了会留下很多废 worktree。

### 3. Git hooks 会一起生效

如果你仓库里有 hooks，需要注意在多个 worktree 下执行体验是否一致。

## 一句话总结

`git worktree` 不是“更复杂的 Git”，而是让你在多分支并行场景里，少犯错、少切换、少打断的一种目录级工作方式。
